# Вопрос по дизайну API: required поля в ответе /api/v1/mobile/sync/push

## Контекст

Мобильный клиент десериализует ответ эндпоинта `POST /api/v1/mobile/sync/push`.
В текущем `documentation.yaml` в схеме ответа (`200`) отсутствует секция `required` —
все поля технически опциональны. Из-за этого клиент вынужден делать все поля nullable,
что усложняет обработку ответа и снижает type safety.

## Вопросы

1. **`result`** — всегда ли поле присутствует в ответе? Если да, добавь его в `required`.

2. **`clientBatchId`** — всегда ли возвращается? Логично ожидать его в ответе для корреляции с запросом.

3. **`RemoteSyncRejected`** (поля `entityType`, `entityId`, `reason`) — если объект rejection
   существует в массиве, все ли три поля гарантированно заполнены? Если да, добавь `required`
   на уровне схемы этого объекта.

4. **`accepted` vs `rejected`** — являются ли они взаимоисключающими (зависят от значения `result`)?
   Если да, стоит задокументировать это через `oneOf` или хотя бы в описании.

## Предложение

Добавить секции `required` там, где поля семантически обязательны — это позволит
мобильному клиенту использовать non-nullable типы и упростит обработку ответа.
