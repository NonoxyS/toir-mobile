# Спецификация мобильного приложения поддержки обхода оборудования

## 1. Общая архитектура

Приложение строится на клиент-серверной архитектуре со **стратегией offline-first**. 

- **Клиент (мобильное приложение)** хранит все ключевые данные в локальной базе (SQLDelight). Работает автономно, синхронизируясь с сервером при появлении сети.  
- **Сервер** (согласно спецификации backend) хранит справочные данные (оборудование, маршруты, чек-листы и т.д.) и принимает результаты обхода.

Основные технологические стеки: Kotlin + Compose Multiplatform на клиенте, Ktor + PostgreSQL на сервере. 

---

## 2. Паттерн и слои приложения

Приложение следует MVVM-модели:

- **View (UI)** – экраны и компоненты Compose.  
- **ViewModel** – бизнес-логика, обработка событий UI, вызовы к репозиториям.  
- **Model/Repository** – доступ к данным (локальная БД + сетевые репозитории).  
- **SyncManager** – модуль синхронизации данных с сервером. 

При запуске приложение загружает локальную БД, затем при первой возможности получает свежие справочные данные с сервера (Bootstrap/обновление конфигурации). Все изменения (результаты проверок, фото, логи) записываются сначала локально и помечаются как «не отправленные», а потом отправляются пачкой на сервер.

## 3. Пользовательские сценарии

### 3.1. Вход в систему

1. Пользователь открывает приложение.  
2. Отображается экран **Авторизации**.  
3. Вводит логин/пароль и нажимает **Войти**.  
4. Приложение посылает `POST /auth/login` (см. спецификацию бэкенда).  
5. При успешной аутентификации приложение сохраняет токен, переходит к главному экрану.  
6. Если токен уже есть и валиден, сразу идёт на главный экран.

### 3.2. Синхронизация данных

- При старте после авторизации выполняется `GET /mobile/bootstrap`: приложение получает список назначений маршрутов, маршруты, точки, оборудование, чек-листы и т.п. Все это сохраняется в локальную БД.  
- В дальнейшем каждые N часов или вручную запускается **синхронизация**:
  - `GET /mobile/config/changes?since=...` – получает дельту изменённых справочников (для обновления оборудования, маршрутов и т.д.).  
  - `POST /mobile/sync/push` – отправка данных обходов, результатов, логов.
  
Локальная БД содержит флаг `sync_status` у каждой записи (pending/synced) для учета синхронизации.

### 3.3. Выполнение обхода

1. **Список маршрутов:** на главном экране пользователь видит доступные ему маршруты (назначения). Для каждого: название, статус (назначен/в работе/завершён), прогресс.  
2. Пользователь выбирает маршрут и нажимает **Начать обход**. Создаётся запись `Inspection` статусом `in_progress`, фиксируется время старта.  
3. Открывается экран **Точки маршрута**: список `RoutePoint`. Для каждой точки показывается привязанное оборудование и статус (не начато/в работе/завершено/пропущено).  
4. Пользователь выбирает текущую точку и нажимает **Открыть**.  
5. Открывается экран **Карточка оборудования**: основные сведения (код, название, тип, локация). Есть кнопка **Открыть чек-лист**.  
6. Пользователь нажимает **Открыть чек-лист**, попадает на экран **Чек-лист проверки** для этого оборудования (см. раздел 4).  
7. В чек-листе пользователь заполняет пункты (см. 3.4). Результаты сохраняются локально.  
8. После заполнения всех обязательных пунктов нажимает **Завершить проверку**. Создаётся запись `InspectionEquipmentResult` со статусом `completed`. Если были пропуски или ошибки – фиксируются их статусы.  
9. Возвращается к списку точек; выбранная точка помечена как завершённая.  
10. Процесс повторяется до последней точки.  
11. После последней точки пользователь нажимает **Завершить обход**. Статус `Inspection` меняется на `completed`.  

Промежуточные данные всегда сохраняются локально; при появлении сети – синхронизируются на сервер (см. 3.2).

---

## 3.4. Заполнение чек-листа

Экран чек-листа состоит из списка пунктов (`ChecklistItem`). Для каждого пункта:

- **Название** проверки.
- **Описание** (если есть).
- **Тип ответа**:
  - **Да/Нет** (Boolean): кнопки или переключатель.
  - **Число**: числовое поле.
  - **Текст**: поле для ввода текста.
  - **Выпадающий список (Select)**: выбор из предопределённых вариантов.
  - **Подтвердить (Confirm)**: просто кнопка «Выполнено».
- **Обязательность** (отмечена иконкой *). Если обязательный пункт не заполнен – не даёт завершить.
- **Фотообязательность**: если требуется фото, рядом выводится иконка «камера».  

При нажатии на пункт «Да/Нет» или вводе значения поле сохраняется немедленно. Добавление фото: возле обязательного пункта появляется кнопка «Добавить фото» – при нажатии открывается камера/галерея. Фото сохраняется как `Photo` и привязывается к текущему `ChecklistItemResult`.

Чтобы завершить чек-лист, пользователь нажимает **Завершить проверку**; проверяются:
- все обязательные пункты заполнены;
- во всех обязательных пунктах с требованием фото фото прикреплены.
В случае ошибки на экране появляется сообщение.

## 3.5. Фотофиксация

Экран фотофиксации вызывается из чек-листа по кнопке «Добавить фото». 

- Пользователь фотографирует оборудование или выбирает фото из галереи.
- Фото сразу сохраняется локально (`Photo` в БД, `sync_status=pending`).
- Экран фотофиксации показывает превью. Кнопки: **Сделать ещё фото** или **Подтвердить**.  
- При подтверждении фото связывается с текущим пунктом чек-листа и возвращаемся к чек-листу.

При синхронизации `Photo` отправляется на сервер через `POST /photos/upload` (см. бэкенд-спецификацию). После успешной загрузки `sync_status` меняется на `synced`.

---

## 3.6. Журнал действий (Action Log)

Скрытый для пользователя лог, но будет использоваться для отладки и аудита. 

Для каждой значимой операции (начало/окончание обхода, открытие точки, изменение результата, добавление фото) создаётся запись `ActionLog` с типом действия, метаданными и временем. Эти записи также синхронизируются с сервером. По требованию могут отображаться на специальном экране «Журнал» (необязательный MVP).

# 4. Описание экранов

## 4.1. Экран авторизации

- **Поля:** «Логин», «Пароль».  
- **Кнопка:** «Войти».  
- При ошибке – текст «Неверный логин или пароль».  
- После входа – автоматический переход к списку маршрутов.  

Должен сохранять JWT-токен.

## 4.2. Главный экран – список маршрутов

- **Заголовок:** «Назначенные маршруты».  
- **Каждый маршрут (карточка):**  
  - Название маршрута.  
  - Статус (Назначен/В процессе/Завершён).  
  - Прогресс (например, «3 из 10 точек»).  
  - Кнопка «Начать» (если статус «Назначен») или «Продолжить» (если «В процессе»).  

При выборе маршрута – открывается экран точек маршрута.

## 4.3. Экран точек маршрута

- **Заголовок:** название маршрута.  
- Список точек маршрута:  
  - Каждая точка показывает:  
    - Наименование оборудования.  
    - Локацию.  
    - Статус: не начато, в работе, завершено, пропущено.  
    - Иконку предупреждения, если по этой точке есть замечания (если были ошибки или пропуски).  
  - Текущая точка выделена.  
- **Кнопки:** «Завершить обход» (активна, когда по точкам есть и завершённые, и незавершённые).  

Нажатие на точку открывает экран **Оборудование/Чек-лист**.

## 4.4. Экран карточки оборудования

- **Поля:** Код, Наименование, Тип, Локация оборудования.  
- **Состояние:** индикатор состояния (например, «Оборудование не проверено/в процессе/проверено»).  
- **Кнопка:** «Открыть чек-лист».  

Нажатие открывает экран чек-листа для данного оборудования.

## 4.5. Экран чек-листа проверки

- **Заголовок:** «Чек-лист проверки».  
- Список пунктов: см. раздел 3.4.  
- **Кнопка:** «Завершить проверку» внизу экрана.  

Каждое действие на экране (ввод ответа, добавление фото) сразу сохраняется локально. При нажатии «Завершить проверку» – валидация заполнения обязательных пунктов.

## 4.6. Экран фотофиксации

- Показывает кадр камеры или выбранное изображение.  
- **Кнопки:** «Сделать фото», «Выбрать из галереи», «Готово».  
- По завершению фото прикрепляется к текущему пункту и экран закрывается.

## 4.7. Экран итогов инспекции

(Опционально, при нажатии «Завершить обход» можно отображать сводку.)

- Итоговый отчёт по обходу:
  - Количество проверенных точек, количество замечаний.  
  - Кнопка «Отправить на сервер» (если синхронизация не активна автоматически).  
  - Статус отправки.

## 4.8. Сервисные экраны

- **Журнал действий:** отображает список записей ActionLog (дата/время, действие, описание).  
- **Настройки/Профиль:** показывает данные пользователя, устройство, состояние БД.  
- **Синхронизация:** индикатор текущего статуса обмена данными.

# 5. Локальная модель данных

В локальной базе (SQLDelight) должны быть таблицы, соответствующие основным сущностям:

- **User, Device, Location, Equipment, Route, RoutePoint, Checklist, ChecklistItem, RouteAssignment, Inspection, InspectionEquipmentResult, ChecklistItemResult, Photo, ActionLog, SyncBatch**.  

У каждой сущности – поля по спецификации бэкенда (см. раздел “Таблицы” в backend-спеке).  

- У каждой записи – флаг `sync_status` (PENDING / SYNCED).  

Например, таблица `inspection_equipment_results` хранит результаты проверки одного оборудования в обходе: ID, inspection_id, equipment_id, status, timestamps. 

Photos хранятся в отдельном хранилище (URI на файл) + метаданные в БД.

---

# 6. Логика синхронизации (SyncManager)

- Приложение собирает «пачку синхронизации» и отправляет на `POST /mobile/sync/push`.  
- Пакет содержит: новые `Inspection`, `InspectionEquipmentResult`, `ChecklistItemResult`, `ActionLog` с их client-generated UUID и `deviceId`.  
- Сервер отвечает, какие объекты приняты/отклонены.  
- После успешной отправки полей запись считается синхронизированной (sync_status=SYNCED).  
- Если ошибка – не отправленные остаются в PENDING для повторной отправки.  
- Отдельно отправка фото: `POST /mobile/photos/upload` (см. backend spec). Photo привязываются по ID пункта чек-листа.

Сервер требует idempotent-поведение: если повторно отправить уже синхронизированный объект, сервер игнорирует дубликат.

---

# 7. API и форматы

Используются согласованные endpoint'ы из спецификации backend:

- **Bootstrap:** `GET /api/v1/mobile/bootstrap` – инициализация после авторизации.  
- **Delta:** `GET /api/v1/mobile/config/changes?since=...` – обновление справочников.  
- **Sync Push:** `POST /api/v1/mobile/sync/push` – передача результатов.  
- **Photo Upload:** `POST /api/v1/mobile/photos/upload` – загрузка фото (multipart).  

Тела запросов и ответов – в формате JSON (описания DTO соответствует backend-спеке). Ошибки обрабатываются по статусу и телу `{code, message}`.

---

# 8. Приемочные критерии

- Все основные сценарии работают полностью офлайн, а при восстановлении сети синхронизируются.  
- На экранах правильно обновляется статус (например, маршрут в работе, точки выполнены).  
- Невозможно завершить чек-лист без заполнения обязательных пунктов.  
- Фото надежно прикрепляются к нужным результатам.  
- После закрытия приложения данные не теряются.  
- При повторном открытии проверка сохраняет прогресс.  
- Интерфейс соответствует стилю: тёмный фон, ясные выделения кнопок, русская локализация.
